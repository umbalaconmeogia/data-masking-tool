# Sample data

The fake values are drawn from name pools under `data-sample/`.

## Locale data format

A **locale** is a directory `data-sample/<locale>-people/` containing two CSV
files with the header `name,kana,romaji`:

| File | Content |
|------|---------|
| `lastname.csv` | Family names |
| `firstname.csv` | Given names |

Columns:

| Column | Meaning |
|--------|---------|
| `name` | The name as written (kanji, Vietnamese with diacritics, …) |
| `kana` | Reading in hiragana; for languages without kana, the plain name again |
| `romaji` | Romanized (ASCII) form, used for email addresses |

Extra columns (e.g. `gender` in the Vietnamese data) are ignored by the tool.

**Adding your own locale** requires no code change: create
`data-sample/english-people/lastname.csv` + `firstname.csv` in this format and
use `english` as the `data_rule` of name rules.

## Bundled locales

| Locale | Rows | Source |
|--------|------|--------|
| `japanese` | 5,000 last + 5,000 first names (kanji, kana, derived Hepburn romaji) | `jp_people_name` table in `test_data_japan.sqlite` (96,805 / 52,133 rows) — see below |
| `vietnamese` | 37 last + 2,536 first names | Plain-text lists in `data-sample/vietnamese-people/raw/` (originally from <https://github.com/umbalaconmeogia/php-test-data/tree/master/src/vietnam>) |

The generated CSVs are committed and are all the tool needs at runtime. The
raw sources (Japanese SQLite files, Vietnamese text lists) are converter
input only and are **not kept in git** — `data-sample/README.md` lists the
download locations, and the converter tools print the same instructions when
the files are missing.

## Regenerating / extending the bundled data

The CSVs are generated by converters in `script-python/tools/`:

```bash
cd script-python

# Japanese: extract from the SQLite source, derive romaji from kana.
# Default keeps an evenly-sampled 5,000 rows per file; --limit 0 exports all.
python tools/convert_japan_name_data.py [--limit 0]

# Vietnamese: from the raw .txt lists (male + female merged, diacritics
# folded to ASCII for romaji).
python tools/convert_vietnamese_name_data.py
```

Rows whose kana cannot be converted to romaji (noise in the source data) are
skipped automatically.

## Storage strategy: CSV vs SQLite

* **Small pools** (name lists, up to a few MB) are stored as CSV —
  human-readable, git-diffable, easy to edit, loaded fully into memory.
* **Large pools** stay in **SQLite** and are queried with indexed random
  access instead of being converted (a CSV export of the Japanese
  address/company data would be ~1 GB). Both Python (stdlib `sqlite3`) and PHP
  (PDO sqlite) can read it natively. The planned `address` data_type will read
  the `jp_address` table (149,102 rows) directly from
  `data-sample/japan-test-data/test_data_japan.sqlite` this way.

## Test data

`docker/test-data/companies.csv` and `employees.csv` hold the demo dataset
(10 companies, 100 employees) used by the docker containers **and** the Python
integration tests. They are generated from
`data-sample/japan-test-data/test_data_demo.sqlite` by
`script-python/tools/generate_test_data.py`, which also regenerates the DDL
init scripts (`docker/mysql-init/`, `docker/pgsql-init/`,
`script-python/tests/fixtures/schema-sqlite.sql`).
